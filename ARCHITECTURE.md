# FinTrack — Architecture

A mobile-first personal finance manager. Single user today, multi-user ready by
construction (every document is namespaced under `users/{uid}`).

## Core design principle: totals are never stored

The original Google-Sheets pain was manually maintaining "Spent Till Now". Here,
**transactions are the only source of truth**. Every total is *derived*:

| Displayed value      | Derived from                                              |
| -------------------- | -------------------------------------------------------- |
| Spent (per category) | `sum(transactions where type=expense, categoryId, month)`|
| Remaining            | `budget.amount − spent`                                  |
| Actual income        | `sum(transactions where type=income, month)`             |
| Emergency fund total | `running sum(emergencyFunds.actual)`                     |
| SIP total invested   | `running sum(sipInvestments.actual)`                     |
| Account balance      | `openingBalance + in − out` over the account's transactions |
| Card outstanding     | `sum(card expenses) − sum(card bill payments)`           |
| Goal saved           | `sum(goal contributions tagged with the goal)`           |

No write ever stores a computed total. Adding/editing/deleting a transaction is
the only mutation; React Query re-derives the rest on the next read.

## Tech stack

- **React 18 + Vite + TypeScript** — strict mode, path alias `@/`.
- **Tailwind CSS + shadcn/ui** (Radix primitives) — themable via CSS variables.
- **Firebase Auth** (Google) + **Firestore** (offline persistence enabled).
- **TanStack Query** — server-state cache, optimistic-friendly invalidation.

## Firestore schema (all under `users/{uid}/…`)

```
users/{uid}                       profile + settings (currency, locale, theme, FY start,
                                  largeExpenseThreshold, accentColor, density,
                                  defaultLandingPage, dashboardLayout)
  categories/{id}                 name, type(expense|income), order, archived, color,
                                  trackDailyPace?
  paymentMethods/{id}             name, order, archived, kind? — now an OPTIONAL "rail" tag
  incomeSources/{id}              name, order, archived
  budgets/{id}                    monthKey, categoryId, amount         ← the PLAN
  transactions/{id}               date, monthKey, amount,              ← SOURCE OF TRUTH
                                  type(expense|income|transfer|cc_payment|goal),
                                  categoryId?, incomeSourceId?, paymentMethodId?,
                                  accountId?, toAccountId?, creditCardId?,
                                  savingsGoalId?, subscriptionId?,
                                  emiId?, recurringRuleId?, isAutoGenerated?,
                                  merchant?, note?
  accounts/{id}                   name, type(bank|cash), openingBalance, institution?,
                                  last4?, color?, order, archived
  creditCards/{id}                name, bankName, last4, creditLimit, billingDay,
                                  dueDay, color, order, archived
  savingsGoals/{id}               name, targetAmount, targetDate?, color?, icon?,
                                  status(active|achieved|archived), order, note?
  subscriptions/{id}              name, service?, amount, frequency(monthly|yearly),
                                  startDate, accountId?/creditCardId?, categoryId?,
                                  autoRenew, status(active|paused|cancelled),
                                  lastGeneratedThrough, note?
  emis/{id}                       name, emiType, monthlyAmount, startDate, months,
                                  endDate, categoryId?, creditCardId?, accountId?,
                                  status, lastGeneratedThrough
  recurringRules/{id}             name, amount, type, frequency, startDate, endDate?,
                                  categoryId?/incomeSourceId?, paymentMethodId?,
                                  creditCardId?, accountId?, merchant?, note?, status,
                                  lastGeneratedThrough
  emergencyFunds/{id}             monthKey, planned, actual, note?
  sipInvestments/{id}             monthKey, planned, actual, kind, name, note?
```

`monthKey` is `YYYY-MM` — the universal index for month and year reporting.

### Relationships

- Transactions hold **soft references** (`categoryId`, `paymentMethodId`,
  `incomeSourceId`). Deleting a category is discouraged in favour of **archive**,
  so historical transactions never orphan. Archived items stay resolvable in
  history but disappear from pickers.
- Budgets join to transactions on `(monthKey, categoryId)` to produce the
  planned-vs-actual view.

## Security & indexes

- `firestore.rules` — a user can only touch documents under their own `uid`.
- `firestore.indexes.json` — composite indexes for the transaction queries
  (`monthKey + date`, `monthKey + categoryId + date`).

## Recurring engine (powers subscriptions, rent, bills AND EMIs)

`src/services/recurringEngine.ts` materialises due occurrences of
`recurringRules` and `emis` as real transaction documents. It runs client-side
once per session (SessionBootstrap) and after rule/EMI mutations.

- **Idempotent**: generated docs use deterministic ids
  (`rec-{ruleId}-{date}`, `emi-{emiId}-{date}`) — re-runs and multi-tab races
  rewrite the same docs instead of duplicating.
- **Atomic**: each batch commits transactions together with the source's
  `lastGeneratedThrough`, so bookkeeping can never drift from the data.
- **Pause/resume**: paused sources are skipped without advancing bookkeeping;
  resume fast-forwards to today so paused periods never backfill.
- Editing a rule/EMI only affects future occurrences — past ones are already
  materialised transactions (which users may edit/delete individually).
- **Subscriptions** are a third engine source: an active, auto-renewing
  subscription materialises each cycle's charge as an expense tagged with
  `subscriptionId` (deterministic id `sub-{id}-{date}`). Non-auto-renew ones
  appear in the renewal calendar as reminders but are never auto-recorded.

## Funding sources & money movement (features 9, 10, 14, 15)

Every transaction's **funding source** is a Bank Account, Cash Wallet
(`accounts`) or Credit Card (`creditCards`). Payment methods are demoted to an
optional descriptive "rail" tag (UPI/NEFT). Five transaction types model all
movement; only `expense`/`income` count toward spending/earning — the rest move
money between the user's own buckets and are excluded from every income/expense
total by construction:

| type         | source / destination                | balance effect |
| ------------ | ----------------------------------- | -------------- |
| `expense`    | `accountId` **or** `creditCardId`   | account ↓ or card outstanding ↑ |
| `income`     | `accountId` (destination)           | account ↑ |
| `transfer`   | `accountId` → `toAccountId`         | from ↓, to ↑ (feature 10) |
| `cc_payment` | `accountId` → `creditCardId`        | account ↓, card outstanding ↓ (feature 15) |
| `goal`       | `accountId` → `savingsGoalId`       | account ↓, goal saved ↑ (feature 8) |

- **Account balance** = `openingBalance + income + transfers in − expenses −
  transfers out − bill payments − goal contributions` (`accountBalance`).
- **Card outstanding** = card expenses − card bill payments (`cardOutstanding`);
  EMI installments are card expenses, so they're included.
- **Goal progress/forecast** (`goalProgress`) projects the average monthly
  contribution forward to estimate a completion date.
- **Subscription MRR/ARR + renewal calendar** (`subscriptionAnalytics`).
- All of the above derive in-memory from a single `listAllTransactions()` fetch,
  so **no new composite indexes are required**.
- **Migration**: every new field is optional. Legacy transactions (no source)
  are "Unassigned"; balances reflect only assigned transactions, and a one-click
  backfill tool on `/accounts` assigns history to a chosen account (batched
  writes). Source is required only for new transactions.

## Large expenses, EMI split, card analytics — derived classifications

- A **large expense** is `amount >= settings.largeExpenseThreshold` evaluated
  at read time — changing the threshold instantly reclassifies all history.
- **EMI vs non-EMI** is `transaction.emiId != null` — derived in
  `splitEmiExpenses`, shown on the dashboard, monthly report and insights.
- **Card stats** (statement-cycle spend, utilization, statement estimate) are
  derived from transactions carrying `creditCardId`, against the card's
  `billingDay` cycle (`currentCardCycle`). Composite indexes:
  `transactions(creditCardId, date)` and `transactions(emiId, date)`.
- Payment-method **kind** resolves lazily (`resolvePaymentMethodKind`):
  explicit field wins, else a name heuristic — zero-write migration.

## Personalization

Accent color and density are `html[data-accent]` / `html[data-density]` data
attributes driving CSS-variable overrides in `index.css`; the default landing
page is honored once per session by SessionBootstrap. All persisted in user
settings with non-breaking defaults.

## Daily-spend pacing

Driven by the system clock: for the *current* month, `daysLeftInMonth` returns
days remaining (inclusive of today), and each expense category derives a "safe
daily spend" = remaining budget ÷ days left. Toggleable per category via
`Category.trackDailyPace` (defaults on). Surfaced per-category in each budget
row. Past/future months show no pacing. (The previous overall "spend up to
₹X/day" strip was removed as misleading.)

## Reports

Two modes (tabbed): **Monthly** (independent month + year filter, summary +
category breakdown + donut) and **Yearly** (income-vs-expense bars, totals,
category breakdown). The monthly filter has its own local state so it never
disturbs the global month selection used by the dashboard/transactions/budgets.

## Future-proofing

New modules (loans, net worth, assets, credit cards, tax) drop in as a new
collection under `users/{uid}`, a service module, data hooks, a route, and a nav
entry — no refactor of existing code. The derived-totals pattern and the generic
Firestore service scale to them directly.
